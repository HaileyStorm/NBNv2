# Nbn.Runtime.WorkerNode

Owns worker node inventory, placement execution endpoints, and lifecycle participation.

## Stable behavior notes

- `WorkerNodeActor` is intentionally split by responsibility: discovery/contracts/state remain separate from placement control flow, hosting/runtime-metadata logic, and shared endpoint/PID helpers so maintenance work can stay file-bounded without changing placement behavior.
- Placement request validation and ack shaping are shared between assignment and unassignment flows inside the placement slice so worker/brain/epoch guardrails stay consistent without changing the wire contract.
- Region-shard placement is artifact-backed. HiveMind now includes runtime metadata hints on placement assignments so WorkerNode can usually skip the IO/Hive metadata bootstrap on the hot path; if usable base artifact metadata is still unavailable at assignment time, WorkerNode returns a retryable `PlacementFailureWorkerUnavailable` ack instead of hosting a synthetic shard.
- Artifact-backed placement resolves non-file `store_uri` values through the shared artifact resolver, including built-in HTTP(S) artifact services and env-mapped logical store URIs. Resolver-wrapped remote stores keep node-local cache behavior under `NBN_ARTIFACT_CACHE_ROOT`; when operators do not override the artifact root/cache env vars, WorkerNode uses the shared user-local app-data default instead of the process working directory so installed-mode artifact-backed shard loads do not fail on read-only install paths. Cache write failures degrade to direct upstream reads instead of failing shard initialization outright, while transport/configuration failures remain retryable placement failures until metadata or artifact bytes become reachable.
- WorkerNode answers targeted peer-latency probe requests for HiveMind placement decisions and reports average RTT plus sample count per requested peer.
- Peer RTT probes use request/reply echo messages against the peer worker root actor. The echo request must respond directly to the caller so remoting preserves the real sender for round-trip measurement.
- Placement reconcile reports preserve worker-root actor ancestry when a hosted assignment PID has already dropped to null, so fallback observed control PIDs still point at the real remote child actor path instead of a naked `address/brain-...` synthetic PID.
- WorkerNode heartbeats publish raw capability snapshots derived from real host probing plus explicit CPU/GPU score calculation, then attach explicit NBN limit metadata (`--cpu-pct`, `--ram-pct`, `--storage-pct`, `--gpu-compute-pct`, `--gpu-vram-pct`; legacy `--gpu-pct` sets both GPU limits).
- Region-shard `auto` backend selection stays conservative by default on this codebase: non-IO shards remain on CPU until they reach the settings-backed `worker.runtime.region_shard_gpu_neuron_threshold` cutoff. The setting accepts an explicit neuron count or `auto`; `auto` now scales a measured baseline threshold (`262144` neurons at GPU score `550.258`) against the worker's effective GPU score instead of hard-coding one flat cutoff for every machine. Operators can still explicitly force `NBN_REGIONSHARD_BACKEND=cpu|gpu`.
- Capability probing and telemetry normalize cache invalidation, metric dimensions, and XML-doc-described public surfaces in one place so maintenance refactors do not silently drift worker inventory or placement observability semantics.
- WorkerNode and RegionHost share the same CUDA-first ILGPU device-selection rules, so worker capability publication, test skip decisions, and runtime shard execution all agree on what counts as a compatible accelerator. OpenCL devices that lack required RegionShard kernel features, including float64 support, are reported as incompatible instead of being advertised as usable GPU accelerators.
- Worker CPU/GPU score microbenchmarks run on initial publication and on the settings-backed refresh cadence requested by HiveMind; routine placement churn does not invalidate the score cache. WorkerNode uses the live CPU/RAM/storage/GPU pressure fields already attached to each heartbeat for placement-pressure updates instead of re-benchmarking on every placement change, and it never bypasses SettingsMonitor with direct capability replies.
