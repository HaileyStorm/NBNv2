# Nbn.Runtime.HiveMind

Owns global tick loop, scheduling barriers, timeout/lateness handling, and brain lifecycle orchestration.

## Stable behavior notes

- `HiveMindActor` shell/state, control-plane registration/authorization/brain-control, settings application/parsing/status, worker inventory/visualization/peer-latency, placement, and tick/reschedule work are intentionally split by responsibility so HiveMind refactors can stay file-bounded without changing runtime behavior.
- `HiveMindActor.ReceiveAsync` is intentionally a thin dispatcher. Message routing stays grouped by responsibility slice so control-plane, artifact, placement, settings, worker-inventory, runtime-surface, and tick/reschedule changes can be reviewed independently without changing behavior.
- Spawn admission and spawn completion are separate control-plane contracts: `SpawnBrain` returns once placement is accepted into the runtime queue with a stable `brain_id`, while `AwaitSpawnPlacement` reports the later placed/ready outcome for that same brain and preserves explicit failure reasons if placement or reconcile fails.
- Placement assignment dispatch is serialized per worker node inside HiveMind at the brain-batch level. HiveMind can dispatch a whole brain's current assignments to one worker together, but only one assignment at a time owns that worker lane until the worker returns its first ack. A later `accepted -> ready/failed` follow-up still belongs to the same placement execution, but it must not block later same-worker assignment dispatches or consume assignment-timeout budget as if the lane were still idle.
- Default spawn-placement wait budgets account for queued worker batches instead of assuming isolated placement. `AwaitSpawnPlacement` without an explicit timeout therefore scales with the current serial-placement window on the targeted workers.
- Placement-assignment retries use the single-attempt assignment timeout for each dispatch, while spawn waits themselves are progress-aware. Accepted assignments are also bounded by that timeout window until they advance to `Ready`, so HiveMind can fail a worker that accepts placement but stalls before hosting completes. The default local-runtime budgets are intentionally short (`2s` assignment, `3s` reconcile) so a bad worker lane fails fast instead of pinning an evaluation batch for tens of seconds. `AwaitSpawnPlacement` without an explicit timeout uses the spawn completion window for the current placement execution, including queued same-worker batches, assignment retry budgets, and reconcile time. Only substantive progress such as shard-ready or reconcile advancement extends that wait again. Immediate `accepted` acks and retry scheduling do not extend the default idle wait on their own.
- Tick dispatch is placement-stability gated. Brains with a non-zero placement epoch do not participate in compute/deliver barriers until their placement execution has completed and lifecycle is `Assigned` or `Running`; shard registrations or same-shard PID replacements that arrive during compute are current registration state for later ticks, not a retargeting of the already-open compute barrier.
- Rescheduling is a real placement transition, not a timer-only delay. HiveMind pauses new ticks, issues placement plan updates, refreshes routing/output-sink state, and resumes only after moved shards re-register with the current placement epoch and opaque assignment id. For placement-managed brains, shard registrations and unregistrations without the active epoch/assignment identity or with a previous epoch/assignment are ignored before they can mutate shard routing state. If a non-recovery reschedule cannot prepare or complete a replacement plan for a brain, HiveMind pauses that brain so stale shards on a lost worker cannot keep poisoning global tick barriers while other newly placed brains continue.
- Whole-brain runtime-state reset is a barrier-coordinated control surface. HiveMind queues one pending reset per brain, rejects duplicates as busy until the pending reset completes, and executes the reset only after that brain has crossed the current `TickDeliverDone` barrier and before the next compute for that brain can begin.
- Actor-driven `UnregisterBrain` from a hosted brain-root/signal-router is ignored while a placement epoch is being replaced or recovered; only stable current-epoch teardown is allowed to remove the tracked brain.
- Reschedule throttles (`reschedule_min_ticks`, `reschedule_min_minutes`) may defer a request, but they do not drop it. Deferred requests are coalesced and retried automatically when the earliest legal window opens.
- Placement scoring prefers low-latency whole-brain locality using inter-worker RTT telemetry. For brand-new brains that fit on one worker, hosted-brain pressure relative to effective worker capacity is considered before exact peer-latency ordering so bursty small-brain spawns spread across eligible workers; current-worker affinity and multi-worker brains remain locality-first.
- HiveMind owns the settings-backed worker capability refresh cadence and sends refresh requests to workers, but refreshed values only enter placement after SettingsMonitor persists the next heartbeat snapshot.
- HiveMind also owns the settings-backed RegionShard auto-GPU cutoff (`worker.runtime.region_shard_gpu_neuron_threshold`) and stamps the resolved per-worker threshold onto placement assignments so WorkerNode can honor the same runtime policy without its own SettingsMonitor subscription. The setting accepts `auto` or an explicit neuron count; `auto` scales the measured baseline cutoff (`262144` neurons at GPU score `550.258`) against each worker's effective GPU score.
- Worker freshness and worker-loss recovery age SettingsMonitor worker snapshots against the SettingsMonitor snapshot clock domain, with local elapsed time only extending the last received snapshot, so mixed-machine placement does not evict healthy workers just because host clocks differ.
- Worker catalog entries are pruned after they disappear from SettingsMonitor snapshots and no tracked brain or worker-dispatch lane references them; if catalog churn exceeds the hard worker-catalog cap, stale untracked entries may also be pruned even when SettingsMonitor still reports them. Freshness filtering still happens in HiveMind against the SettingsMonitor clock domain, and active placement/tick-barrier state is never pruned as cache.
- Placement and rebalance treat GPU compute score and VRAM fit as separate constraints, but only weight GPU placement when RegionShard GPU execution is enabled (`NBN_REGIONSHARD_BACKEND` not forced to `cpu`); `auto` backend selection still chooses the stronger scaled backend per shard on the worker rather than hard-coding GPU-first behavior. CPU/RAM/storage pressure always blocks generic placement, while GPU VRAM pressure only blocks generic placement/rebalance when the worker has no viable CPU fallback path; a GPU-preferred shard may therefore fall back to CPU placement instead of failing purely because VRAM is hot.
