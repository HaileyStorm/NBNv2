# Nbn.Tools.Workbench

Owns desktop orchestration/visualization/debug UX and deterministic view-model dispatch behavior.

## Stable responsibilities

- Host operator workflows over IO/debug/viz and reproduction surfaces without exposing actor placement details.
- Keep Workbench boundaries explicit: `ShellViewModel` owns reconnect/status fan-out, `OrchestratorPanelViewModel` owns explicit local launch/stop flows and launch diagnostics, and `WorkbenchClient` remains the transport/request boundary without auto-starting runtime services on connect.
- Keep panel ownership explicit inside Workbench: IO/Reproduction/Speciation/Debug view-models own their panel-specific runtime behavior, settings mapping, and confirmation semantics, while shared picker/export helpers stay utility-only and never own transport or policy decisions.
- Keep command/view-model correctness independent of Avalonia loop availability (dispatcher wrappers execute inline when no UI lifetime exists).
- Keep visualization semantics deterministic across rendering modes.
- Keep visualization ownership split by concern: `VizPanelViewModel.Activity` owns event/cadence/filtering flow, `VizPanelViewModel.Projection` owns versioned projection/layout refresh, `VizPanelViewModel.CanvasInteraction` owns selection/hover/pin/hit-testing state, and `VizActivityProjection*` / `VizActivityCanvasLayout*` remain pure builder helpers.
- Treat `DebugHub` and `VisualizationHub` as Observability-owned actor names resolved at the configured Observability host/port (not Workbench-owned services).
- Surface Observability connection state from runtime reachability/availability signals, not from endpoint text configuration alone.
- Keep `Settings` host/port/name as the only operator-configured endpoint coordinates in Orchestrator; all other service endpoints/actor names are discovery-driven from SettingsMonitor (`service.endpoint.hivemind`, `service.endpoint.io_gateway`, `service.endpoint.reproduction_manager`, `service.endpoint.speciation_manager`, `service.endpoint.worker_node`, `service.endpoint.observability`) and shown as read-only endpoint displays.
- Orchestrator `Local Launch` exposes the local advertise-host setting for this desktop. It is seeded from the resolved default advertised/LAN host, applies to the Workbench receiver plus all locally launched runtime/services/tooling flows, and can still be cleared when the operator explicitly wants runtime auto-advertise selection instead.
- Drive Orchestrator endpoint chip online/offline state for non-Settings services from SettingsMonitor discovery evidence (endpoint identity + fresh node heartbeat visibility), not from direct client transport connection status alone.
- Keep Speciation represented in all Orchestrator control surfaces that manage core runtime services: Endpoints status rows, Local Launch start/stop controls, and `Start All` / `Stop All` orchestration flow.
- Launch runtime services only from explicit Local Launch commands (`Start` / `Start All`); connection attempts never auto-start runtimes.
- `Start All` is fail-fast for the current invocation: if one launch step fails after earlier local launches succeeded, Workbench stops the services it started in that `Start All` run instead of leaving a partial local stack running.
- Local Launch uses SettingsMonitor as the only required cross-service bootstrap coordinate for non-Settings services and remains port-only per local process start; Workbench does not inject static peer host/port wiring between runtime services.
- Local Launch treats discovered service hosts as runtime connectivity coordinates, not as unconditional self-advertise values for new local processes: launched runtime services bind all interfaces by default, reuse an explicit `--advertise-host` only when that host resolves to this machine, and otherwise let runtime remoting resolve a reachable local advertised address so mixed local/remote topologies do not mis-advertise a local service as the remote host.
- On supported platforms, Local Launch best-effort opens local inbound firewall access for the launched runtime TCP port; permission or tooling failures surface as launch warnings instead of silently leaving the service LAN-inaccessible.
- The Workbench operator client receiver also binds all interfaces by default so remote Settings/IO/HiveMind/Observability request-response flows can route replies back to the desktop client from other machines.
- Workbench has two launch modes. In local source-checkout mode, Local Launch preflights the target project through a Release build before process start, then runs the freshly built executable (or a post-build `dotnet run --no-build` fallback if no executable output is present). In installed-binary mode, Workbench must never build source projects or call `dotnet run`; it resolves installed sibling binaries from the shipped runtime manifest and falls back to PATH aliases only when the manifest is unavailable. Workbench must not silently reuse stale existing Release outputs after source edits.
- Orchestrator exposes a `Profile Current System` action that launches the standalone perf-probe tool against the currently configured SettingsMonitor coordinates. The probe discovers current runtime endpoints through SettingsMonitor rather than trusting stale manual endpoint text, writes report artifacts outside the running services, and does not auto-start any runtime service as a side effect.
- Orchestrator exposes a dedicated settings-backed `Worker Policy` surface for worker capability refresh cadence, pressure-rebalance thresholds, and the RegionShard auto-GPU neuron cutoff (`worker.runtime.region_shard_gpu_neuron_threshold`), so operators can tune those system-wide values without hunting through the generic Settings card and without depending on Workbench remaining open.
- Orchestrator also exposes a read-only `System Load` summary that combines SettingsMonitor worker capability snapshots with HiveMind status: aggregate CPU/RAM/storage (and VRAM when available) usage versus configured worker quotas, current over-quota worker count, recent worker-pressure count across the active rebalance window, and recent tick timeout/lateness plus automatic cadence-reduction state.
- Debug mirrors that `System Load` summary so operators can inspect worker resource usage and current/recent pressure while staying on the debug stream surface; it reuses the same SettingsMonitor worker inventory plus HiveMind status path instead of maintaining a separate telemetry source.
- Orchestrator `Settings` can pull a source snapshot from another running SettingsMonitor by host/port/name and apply the imported values into the currently connected SettingsMonitor, but discovery-owned `service.endpoint.*` rows are never copied. Pull failure diagnostics distinguish TCP reachability problems (for example loopback-only bind or firewall blockage) from actor-name/remoting mismatches, and same port numbers on different machines are valid.
- Speciation operator workflows live in a dedicated Speciation pane: service status, Settings-backed policy editing, epoch-aware memberships/chart views, epoch deletion/reset controls, and explicit confirmed `Start New Epoch` action. Service start/stop remains owned by Orchestrator.
- Speciation pane exposes a single epoch selector that applies to counts, trend cards, flow, split-proximity, and cladogram views together, while separately summarizing how many epochs exist and which epoch is currently active; `All epochs` remains the default overview mode and current active epoch is called out explicitly in status text.
- Speciation pane also hosts EvolutionSim session controls (`start` / `stop` with automatic status polling) for standalone reproduction/speciation stress runs, without turning Workbench into a runtime-service owner.
- Speciation pane simulator status uses source-separated similarity labels (`sim_overall`, `sim_assess`, `sim_repro`, `sim_commit`) so operators can distinguish compatibility-check minima from reproduction/commit minima and compare simulator inputs against Speciation history cohesion/split-pressure charts. `sim_commit` reflects the committed source-species similarity reported back by Speciation when available, with deterministic fallback to legacy pre-commit similarity only for older or missing runtime metadata.
- Speciation pane visualizes taxonomy dynamics with two live-updating history charts derived from `SpeciationListHistory`: a per-species population line chart and a top-to-bottom stacked flow chart where horizontal width denotes relative population share (0-100%).
- Flow-chart lineage placement is lineage-family-aware across actual samples: derived species are allocated inside their nearest visible source-parent family span (falling back up the lineage chain when direct parents are not charted), and when a parent species still has its own population the parent ribbon splits around the centered child branch so the new lineage opens from the middle of that visible family span. Parent split-halves bridge from the prior unsplit edge when a child first appears so the chart does not leave background wedges at the branch origin. Root/family left-to-right order stays pinned to first appearance order instead of re-ranking when later population leaders change, and flow areas render fill-only without darker per-species outline strokes. Species do not receive synthetic pre-origin segments; nothing is drawn before first actual appearance. Hovering a visible flow ribbon reports the species plus the nearest sample's population/share and shows a matching color swatch, and the dedicated pop-out window reuses the same chart source/selection logic instead of maintaining a separate rendering path.
- Population history chart keeps raw counts in legend/tooltips but renders y-position on a logarithmic transform (`log10(1+count)`) so early growth and low-population species remain visible at the same time as high-population species.
- History-chart semantics are epoch-aggregated when loaded history spans multiple epochs; when loaded history has a single epoch with multiple rows, charts switch to within-epoch row sampling so trend lines show progression instead of collapsing to one x-sample. Workbench still pages/load-combines full history for status and cladogram, while the visible `Chart window` textbox is a trend-only recent-row cap (`0 = all loaded history`) for the Epoch Trend card.
- Speciation pane no longer retains or renders a raw Speciation History table; it keeps only the aggregate chart/cladogram/epoch-summary state needed for the visible cards.
- History/chart refresh is automatic and background-driven: connection/discovery changes trigger an initial load, then a lightweight polling loop refreshes status/history continuously, refreshes membership counts on a slower cadence, and avoids rewriting Settings on each poll.
- Population and flow charts cap species cardinality for readability/perf: population lines show the top 12 species by population, and the inline flow chart normally shows up to 11 visible species plus an aggregated `Other species` bucket, except when the total taxonomy fits the chart's full `visible + Other` capacity (for example exactly 12 inline species, 24 expanded species, or 48 wide-expanded species), in which case Workbench shows the real species instead of collapsing the last one into `Other`. Flow selection now defaults to mixing founder-family population leaders with newer species (`Include newest` toggle on) while preserving founder/root balancing across multiple lineages; the newest-side ranking prefers species with population `> 2` and only backfills lower-population newcomers when that bucket would otherwise have spare capacity. Disabling that toggle reverts to the previous population-only descendant ranking. The maximized flow pop-out raises the visible cap to `23 + Other`, or `47 + Other` on wide (`>= ~1700px`) windows, and recomputes layout responsively on resize.
- Single-epoch/single-sample line-series still render a visible tick segment (instead of a non-drawable move-only path), so split-proximity and population trend cards remain visually truthful when only one epoch sample is loaded.
- Speciation pane `Epoch Trend` labels separate cohesion from split pressure: `Max within-species divergence` reflects per-membership assigned-species similarity extracted from `decision_metadata_json` (prefers `lineage.intra_species_similarity_sample`, then `lineage_assignment_similarity_score`), while split-source similarity remains available separately in metadata.
- Speciation pane includes a split-proximity trend chart showing the most recent 12 species with split-proximity samples, plotting minimum `(source-species similarity - effective split threshold)` per epoch and running minima across loaded rows in single-epoch sampling mode so operators can track which active species are nearest to split-trigger conditions (`<= 0`). New runtime metadata exposes assigned-species split proximity as the generic field and source-species split pressure under explicit source-prefixed keys, so chart parsing prefers runtime-provided source-prefixed split-proximity deltas and source dynamic thresholds when available, then falls back to older generic/source metadata plus policy threshold-minus-guard. Split-proximity chart y-position uses a signed-log transform to keep near-threshold movement readable while preserving plus/minus sign, but its visible bounds now follow the actual observed min/max values instead of forcing symmetry around zero.
- Speciation pane includes a lineage cladogram view generated from history `decision_metadata_json` (prefers `lineage.source_species_id`, falls back to `lineage.dominant_species_id` for older rows) plus `lineage_diverged_new_species` decisions, providing an operator-facing parent -> derived species relationship summary directly from committed lineage evidence.
- Speciation history visualizations compact derived-species labels by surfacing the runtime lineage code suffix from display names (for example `[A]`, `[AB]`, `[BD]`) instead of verbose root stems, while numbered root species themselves remain visible as full names such as `Unclassified-1` / `Unclassified-2`. Chart legends omit redundant live `now ...` value text so large taxonomies remain readable, and species colors are assigned deterministically from first appearance order with a recent-window similarity guard so the shared population/flow/split-proximity/cladogram color map does not drift into visible same-hue runs. Legend swatches in the speciation charts are interactive: clicking a species swatch opens a compact light-dismiss preset palette, and the selected override applies immediately across the shared chart/cladogram color map for both the inline pane and the flow pop-out.
- Speciation pane policy editor surfaces anti-fragmentation/re-alignment knobs from `assignment_policy`: `lineage_split_guard_margin`, `lineage_min_parent_memberships_before_split`, `lineage_realign_parent_membership_window`, `lineage_realign_match_margin`, `lineage_hindsight_reassign_commit_window`, and `lineage_hindsight_similarity_margin`.
- Workbench default draft values for lineage splitting are conservative but split-reactive (`match=0.92`, `split=0.88`, `lineage_hysteresis_margin=0.04`, `split_guard_margin=0.02`, `min_parent_memberships_before_split=1`, `parent_consensus_threshold=0.70`) and remain user-editable before `Apply Config`.
- Speciation pane speciation-policy draft values are SettingsMonitor-backed via `workbench.speciation.*` keys (load on connect, update on setting-change events, persist on config/limit updates); the pane does not require a manual `Load` action. Simulator-only controls remain tool-local and are not persisted to Settings DB.
- Speciation pane simulator controls expose `run_pressure_mode` (`divergence|neutral|stability`) and `parent_selection_bias` (`divergence|neutral|stability`) so run-count pressure and parent-selection lineage/breadth pressure can be tuned independently per session; pane defaults to `run_pressure_mode=divergence` and `parent_selection_bias=stability` with exploratory run-count bounds (`min_runs=2`, `max_runs=12`).
- Simulator parent seeding automatically materializes the effective parent pool from selected Parent A/Parent B brains, optional override files that replace those slots, and additive extra parents from dropdown/file import; launcher enforces at least two distinct total parents and emits one `--parent-brain` argument per distinct live parent so multiple live instances of the same exported brain remain separate founders in simulator/speciation runs.
- Treat spawn placement and visualization readiness as separate gates: Workbench waits for explicit IO/HiveMind placement readiness first, using the runtime-provided placement budget rather than a short local deadline, then separately waits for the spawned brain to appear in runtime brain/lifecycle views before considering it visualization-ready.
- Drive worker endpoint visibility from SettingsMonitor heartbeat/registration data as a multi-worker list (`active`, `degraded`, `failed`) with transient retention before removal; include compact per-host brain hints from placement/controller ownership when available, and do not infer worker availability from local endpoint text fields alone.
- When Workbench evaluates node/controller freshness from remote SettingsMonitor data, it uses the SettingsMonitor time domain (worker snapshot clock or latest returned heartbeat times), not the local desktop wall clock, so mixed local/remote runs do not blank hosted actors or active-brain lists under cross-machine clock skew.
- Continuous input-vector mode is tick-driven from selected-brain output callbacks (scalar and vector, with selected-brain + tick de-duplication), not `VizTick`, so input replay remains tied to real simulation ticks even when visualization stream cadence is throttled.
- Visualizer `Inputs` card includes an optional `Randomize after send` toggle for vector sends; when enabled, manual `Send Vector` and continuous per-tick sends both regenerate the draft vector after each successful send while preserving width validation and existing tick-driven dispatch semantics.
- Visualizer `Inputs` and `Vector Outputs` cards surface system IO coordinator policies backed by SettingsMonitor (`io.input_coordinator.mode`, `io.output.vector_source`); UI selection writes canonical values to SettingsMonitor and consumes setting-change feeds so controls stay synchronized with external updates and Orchestrator settings workflows.
- Visualizer cadence controls are SettingsMonitor-backed: tick cadence (`tick.cadence.hz`) and visualization cadence (`viz.tick.min_interval_ms`, `viz.stream.min_interval_ms`) are written via settings and reflected back through settings snapshots/live change feeds, including direct Settings DB value updates detected by SettingsMonitor for existing setting rows, so Orchestrator `Settings` remains the single live source of truth; Workbench enforces operator ranges (`tick: 1-1000ms`, `viz: 100-3000ms`) and prevents visualization cadence from being faster than the current tick cadence.
- When external tick cadence changes arrive, Workbench refreshes HiveMind status so the operator surface keeps the configured cadence target distinct from the current authoritative runtime target instead of collapsing both into the local draft textbox.
- Reproduction pane defaults/current controls are SettingsMonitor-backed via `repro.config.*` keys; Workbench applies settings-sync snapshots and live setting-change events to keep Reproduction knobs aligned with Settings DB state, while Orchestrator `Settings` remains the only pane that intentionally preserves dirty local edits against newer authoritative values.
- Workbench definition-topology hydration consumes runtime-exported `ArtifactRef` values through the shared artifact resolver for non-file `store_uri` values, so Viz can load definitions from registered stores, direct HTTP(S) artifact-service URIs, or `NBN_ARTIFACT_STORE_URI_MAP`-bootstrapped logical store URIs instead of assuming every artifact lives under a directly reachable local directory. File-backed flows may still probe known local artifact roots for operator-created stores, and resolver failures remain surfaced as actionable operator errors rather than silent local-path fallback.
- Designer supports artifact-store-first definition/snapshot round trips: operators can store local `.nbn` / `.nbs` bytes into a chosen artifact store (local path, `file://`, registered logical non-file URI, env-mapped non-file URI, or direct HTTP(S) artifact service), immediately see the resulting `ArtifactRef` fields (`sha256`, `media_type`, `size_bytes`, `store_uri`), reopen `.nbn` / `.nbs` documents by artifact reference without exporting local files first, and hand `.nbn + .nbs` artifact refs back to IO/HiveMind for runtime-owned restore. Unresolved non-file `store_uri` values fail explicitly through the shared resolver path; Designer does not silently translate them into local paths, and Workbench does not own live snapshot freshness or partial shard restore behavior.
- Workbench local-origin artifact flows publish through a reachable local HTTP artifact store before remote runtime calls. Designer brain spawn, Reproduction file-based parent uploads, and artifact-restore requests that resolve local or process-local stores all hand IO/HiveMind/Reproduction `http://...` `ArtifactRef`s instead of assuming shared local filesystem roots or shared in-process store registrations between the operator desktop and runtime hosts. That reachable store reuses a stable port adjacent to the Workbench client port (default `12091`) so cross-machine firewall allowances can be configured once instead of chasing ephemeral spawn ports. Workbench only auto-promotes refs it can resolve locally; opaque runtime-exported refs already backed by remote runtime storage remain pass-through.
- Designer brain spawn includes explicit input/output widths alongside the published base-definition `ArtifactRef`, so placement planning does not depend on an immediate successful re-read of base definition metadata just to recover the IO-region spans.
- Speciation-pane EvolutionSim launch includes Settings endpoint coordinates so simulator reproduction requests use the same `repro.config.*` settings profile (not ad-hoc local defaults), and Workbench surfaces local firewall attention for the simulator remoting port because IO/Reproduction replies must be able to reach the simulator process on that inbound port.
- Random brain generation applies activity guardrails in non-fixed modes: input-region neurons are normalized to a signal-responsive profile (`ACT_IDENTITY`, `RESET_ZERO`, `ACCUM_SUM`, zero thresholds), internal neurons use a bounded/stable activation subset (`ACT_IDENTITY|ACT_STEP_*|ACT_CLAMP|ACT_SIN|ACT_TANH|ACT_SIG|ACT_PCLAMP`) with tighter threshold caps (pre-activation `<=36`, activation `<=40`) and parameter clamps near decoded `[-1,+1]`, output-region neurons use bounded defaults (`ACCUM_SUM`, activation subset `ACT_CLAMP|ACT_TANH|ACT_SIG`, tighter threshold caps) and non-fixed I/O reset behavior is pinned to `RESET_ZERO`; non-fixed internal reset selection uses a curated fading-memory subset (`RESET_HOLD` 35%, `RESET_HALF` 30%, `RESET_TENTH` 20%, `RESET_HUNDREDTH` 10%, `RESET_ZERO` 5%), non-fixed accumulation weights de-emphasize `ACCUM_PRODUCT`, `ACCUM_NONE` is excluded from random weighted picks, generation biases dense intra-region / sparse inter-region topology, and generation seeds an input-to-output influence path plus high-likelihood output->input-endpoint recurrence bridges (outside region `0`) with low target sharing preference.

## Visualization layout decisions

- Baseline full-brain map uses axial 2D layout.
- Optional `Projected 3D (R&D)` mode is allowed for region-map rendering only; focus-mode neuron layouts remain 2D.
- Projected mode uses axial slice depth (`z=-3..+3`) and deterministic lane offsets, then clamps to canvas safe bounds.
- Projection must fall back to axial 2D when region data is invalid/empty, geometry is non-finite, or spread is below minimum usability thresholds.
- Hover/select/pin/tooltips and keyboard navigation operate on final 2D coordinates and existing spatial indexing regardless of projection mode.
- Focus-mode gateway placement applies deterministic de-overlap search so gateway region nodes do not stack on top of each other.
- Viz canvas empty-space double-click resets camera to default center; when already at default center in region-focus mode, it returns to full-brain view.
- Layout cost for projected region-map mode is `O(region_count)` and bounded by NBN region count (`<= 32`); no GPU dependency is introduced.
- Viz panel exposes an optional compact top-N activity chart as an in-canvas top-right overlay (below canvas toolbar/status text): full-brain mode ranks regions with score `1 + |value| + |strength|` and log y-axis (`log(1+score)`), while focus mode ranks neurons in the focused region (including output region `31` neurons when focused) using signed `VizNeuronBuffer` values on a linear y-axis; chart history range is configured in seconds (default `~3s`) then converted to a bounded tick window from current cadence, and chart/canvas projection history uses a larger internal retention buffer than the visible stream table to avoid short-window resets under high event throughput.
- Buffer monitoring uses sparse `VizNeuronBuffer` events (initial sample plus value-change updates) and chart projection carries forward last-known neuron buffer samples across global tick progression, so low-activity windows keep advancing on the x-axis without flooding unchanged samples.
- Canvas color modes include two energy-focused variants that preserve the existing colorblind-safe blue/orange paradigm: `Energy: Reserve` uses latest buffer/value reserve sign+magnitude, and `Energy: Cost pressure` visualizes estimated burn pressure from activity/recency/fanout relative to reserve.
- Viz canvas color transfer supports both `Linear` and `Perceptual log/symlog` curves; perceptual mode uses `log1p` for non-negative activity-like signals and symmetric log for signed value/reserve signals to improve low-mid contrast without flipping sign semantics.
- Color-mode and transfer-curve selectors expose detailed tooltip legend text with concrete palette meanings (for example, warm orange positive vs cool blue negative) so operators can interpret node/edge color intent without reading implementation details.
- Default/fit recenter operations apply a leftward visual bias while the mini chart overlay is enabled so core graph content remains readable under the top-right overlay.
- Focus-mode concentric/gateway placement and right-side region lanes apply a deterministic avoidance bias for the top-right overlay zone to reduce node overlap near the chart.
